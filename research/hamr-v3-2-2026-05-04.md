# HAMR v3.2 — post-spike redesign baseline

## 0. Document role

This is the **HAMR design baseline after** the 6-spike validation gate
(EPIC #860). It supersedes `research/hamr-v3-2026-05-04.md` (the v3
spike-input design) and incorporates:

- 6 spike findings (S1 #876, S2 #877, S3 #878, S4 #879, S5 #880, S6
  #881).
- 3 client questions answered post-gate-review (Q1 OS-agnostic key
  store; Q2 judge quorum independence; Q3 failover & redundancy
  explicitness).
- 8 cross-cutting remediations (R1–R8).
- 1 new shared module (`scripts/global/judge-quorum.js`).
- A 5-wave execution order across the revised 9-child MVP list.

Substrate-level design (CF Worker + R2 + KV + MCP + Tailscale fleet)
**survives unchanged**. What changes is the trust model, latency
contract, cache strategy, compression gate, key-storage model, judge
gate, and the explicitness of failover.

## 2. What survives unchanged from v3

- Cloudflare Worker + R2 + KV + MCP front-end as the centralised
  substrate, with Wrangler 4.x build pipeline.
- Per-tier sub-bundles: `fim-5kb`, `routing-12kb`, `governance-30kb`,
  `architect-90kb`. S5 confirmed the 32% compression floor at
  100% rule-coverage; sub-bundle granularity remains correct.
- 9-row per-provider native-caching matrix (Anthropic / OpenAI /
  Gemini / Groq / Cerebras / OpenRouter / Ollama / vLLM / llama.cpp).
- Tailscale fleet routing for the local-fleet lane via existing
  `cascade-dispatch.js` + `litellm-client.js` (REFACTOR per S1 #876).
- npm-bundled snapshot + GitHub release asset CDN as the 3-tier
  bundle distribution fallback (v2 carryover, validated by S2 #877's
  capability-probe baseline).

## 3. Eight remediations (R1–R8)

### R1 · Trust model: signed governance state with 4-tier key store

**Source evidence:** S6 #881 A3-E HIGH residual (poisoned fleet model
fabricates `CONSULTANT_CLOSEOUT`); Q1 (OS-agnostic key storage).

**Architecture:** every machine-emitted baton handoff artifact
(`MANAGER_HANDOFF`, `COLLABORATOR_HANDOFF`, `ADMIN_HANDOFF`,
`CONSULTANT_CLOSEOUT`, `BLOCKER_NOTE`, `ADR_*`, `BUNDLE_PUBLISH`,
`MAILBOX_ENVELOPE`) carries a trailing block:

```
signature: <ed25519-sig-base64>
key_id: <operator-key-id>
timestamp: <iso-8601>
```

`label-lint.yml` extends to verify the signature against the
operator's published keyring before accepting any `status:*` or
`role:*` label transition. Unsigned or invalid-sig artifacts fail
the gate.

**4-tier OS-agnostic key store** (Q1 redesign):

| Tier | Storage | Available on | Notes |
|---|---|---|---|
| T1 | Hardware enclave | macOS Secure Enclave, Windows TPM 2.0, Linux TPM 2.0 | Auto-detected; strongest |
| T2 | OS keychain | macOS Keychain, Windows Credential Manager, Linux libsecret/KWallet/GNOME-Keyring | `keytar` |
| T3 | Age-encrypted file | Any OS | Passphrase or YubiKey or PAM |
| T4 | Ephemeral in-memory | Any environment incl. CI | Short-lived only |

`hamr:doctor` (#896) probes T1 → T2 → T3 → T4 and reports the
achieved tier. **No OS is restricted.** ChromeOS LXC, BSD, illumos
all map cleanly to T2 or T3. The 4-h JWT TTL fallback applies only
when T4 is selected (CI runners). On T1–T3 the operator key is
durable and the JWT TTL extends to a 24-h policy default.

**New module:** `scripts/global/baton-signing.js` (#894) implements
sign/verify over canonical JSON (RFC 8785 JCS) with the 4-tier
probe. Existing `scripts/global/agent-signature.js` extends from
textual provenance to cryptographic signing.

**Judge gate** is folded into R1 but described in detail in §3.2
below; the wording "non-fleet cloud judge" from the v3.1 sketch is
**deleted**.

### R2 · Mailbox: signed envelopes with replay protection

**Source evidence:** S6 #881 A1-T MEDIUM (R2 mailbox writes
unauthenticated and replayable).

Each A2A envelope (Google A2A spec carryover from v2) carries:

- `headers.signature` — Ed25519 over canonical JSON.
- `headers.nonce` — UUIDv7 (timestamp-ordered, for replay-set
  bookkeeping).
- `headers.expires_at` — ISO-8601, ≤24 h from issue.

Worker `/mailbox/read` (HAMR child 5) verifies signature against
publisher-keyring KV; rejects unknown publishers, expired envelopes,
and replayed nonces (KV-tracked nonce set with 24-h TTL).

Worker `/mailbox/write` requires bearer DPoP token + Ed25519
envelope-signature; logs publisher-keyhash + nonce to Tail Workers
for forensics.

### R3 · Bundle integrity: SLSA-L3 verification before consumption

**Source evidence:** S6 #881 A1-E + A4-T MEDIUM (Worker-side bundle
tampering and supply-chain attacks on published HAMR bundles).

Build path: `slsa-framework/slsa-github-generator` produces SLSA-L3
attestation; `sigstore/cosign` produces Cosign Bundle 1.0; both
attached to the GitHub release asset and to `bundles/<sha>.tar.zst`
in R2.

Consumption path: `hamr:doctor` (#896) runs `slsa-verifier
verify-artifact` against the active bundle before reporting
`hamr ok`. MCP clients refuse to connect to a Worker advertising a
bundle with no valid SLSA attestation.

CI gate: PR cannot publish a new bundle without SLSA + Cosign
artifacts — `release-please` pipeline extended.

**TruffleHog (AGPL-3.0)** retained as a subprocess-only invocation
(no direct dep, AGPL distribution boundary preserved per S6 #881).

### R4 · Latency contract: warm-only ≤80 ms claim

**Source evidence:** S3 #878 measured cold paths 108–116 ms p50
exceeding the v3 ≤80 ms claim by 28–36 ms.

- Stated overhead budget: **≤80 ms p50 / ≤120 ms p95** on
  warm-connection cache-hit path.
- Cold-path budget: **≤180 ms p95** disclosed transparently.
- HTTP/2 keepalive REQUIRED on every operator → Worker connection.
  Connection-pool persistence in `state-offload-client.js` (REFACTOR
  per S1 #876) and the new HAMR client SDK.
- KV edge-cache MANDATED via `Cache-Control: public, max-age=300,
  stale-while-revalidate=60` on read-mostly endpoints
  (`/bundle/<profile>/<sha>`, `/wiki/index`, `/healthz`).
- `npx megingjord init` sample output revised from "60 ms" to actual
  measured warm-path numbers (54 ms p50 / 80 ms p95 per S3).
- penguin-1 path (DERP relay, 64 ms p50) flagged in `hamr:status` as
  degraded; HAMR scheduler down-weights it for latency-sensitive
  routes.

### R5 · Cache strategy: 1-h extended cache default + rebuild rate limit

**Source evidence:** S4 #879 found ephemeral 5-min cache misses on
baton transitions (Manager → Collaborator → Admin → Consultant
typically spans 15–60 min).

- Default cache mode: 1-hour extended (`cache_control: { type:
  "ephemeral", ttl: "1h" }`) for Sonnet 4.6 and Opus 4.7. Falls
  back to 5-min ephemeral on Haiku 4.5 (extended cache may not yet
  be GA on Haiku — re-verify before MVP execution).
- Bundle-rebuild cadence rate-limited to ≥5 minutes at the Worker
  layer. A signed mailbox arrival (R2) does NOT trigger immediate
  rebuild; the Worker batches arrivals and rebuilds at most once
  per 5 min.
- Tool definitions live in cached prefix (system block), never in
  the per-call `messages` tail. Enforced by `cascade-dispatch.js`
  builder (REFACTOR per S1 #876).
- Cache-hit-rate gate: `hamr:quota` exposes the rolling 7-day hit
  rate; if it drops below 80%, the dashboard fires a warning panel
  and the routing engine considers wider cache TTL.

### R6 · Compression: two-stage rule-coverage gate

**Source evidence:** S5 #880 measured 100% rule-coverage at upper
bound for both deterministic top-k and Cerebras llama3.1-8b
methods; flat 100% curve is keyword-grounded grading bias.

- **Stage-1 (every bundle build):** deterministic top-k extractive
  over a 47-keyword governance vocabulary. Threshold: 100% keyword
  presence (any miss = build fail). **Replaces LLMLingua-2 as the
  production first-pass** — zero LLM cost, fully reproducible.
- **Stage-2 (weekly, on rule-change PR, on major refactor):**
  60-question reasoning-grounded quiz including counter-factual /
  boundary cases (filed as #893). Graded by judge-quorum
  (`scripts/global/judge-quorum.js` #895). Threshold ≥97%.
- Compression target: 32% of source size (≈68% tokens saved). Below
  that, both methods evict rule-bearing paragraphs.
- LLMLingua-2 retained as research candidate but no longer the
  production path.
- N=5 stochasticity check on Stage-2 grader before any new
  compression pipeline ships.

### R7 · Capability-gated deployment + 3-tier degraded mode

**Source evidence:** S2 #877 capability probe shows operator
environment: CF authenticated; R2 not enabled; Wrangler not
installed; MCP SDK not installed; npm not authenticated.

- `hamr:doctor` (#896) is the entry point. Reads
  `.dashboard/capabilities.json` and emits per-capability remediation
  plans with copy-paste install commands.
- **Three-tier degraded mode** (Q3 explicit):
  - **Tier 1 (full HAMR):** all 6 capabilities + judge quorum
    reachable.
  - **Tier 2 (degraded):** CF reachable but ≥1 capability missing →
    read-only HAMR (fetch-only bundles from CDN/npm fallback; no
    mailbox writes; deterministic governance gates only).
  - **Tier 3 (offline / pre-HAMR):** CF unreachable → harness
    operates as it does today (cascade-dispatch + free-router +
    capability-probe + wiki tooling).
- No `npx megingjord init` step silently deploys paid resources.
  Any step that would create an R2 bucket, deploy a Worker, or
  enable a paid feature requires explicit `--accept-paid-resources`.

### R8 · Live-measurement closure plan

Three deferred follow-ups close residual validity gaps from spikes:

1. **Wave 1 #891 (S3 live deploy):** 1-day operator-authorized run.
   Tightens warm-path RTT from "derived" to "measured."
2. **Wave 1 #892 (S4 live cache):** ≤$0.50 operator-authorized run.
   Validates 80% hit-rate assumption; expected spend ~$0.07.
3. **Wave 1 #893 (S5 Stage-2 quiz):** 60-question counter-factual
   quiz graded by judge-quorum. Validates rule-coverage beyond
   keyword-presence layer.

These are NOT MVP children — they're validation closures filed as
separate tickets.

## 3.2 Judge-gate redesign (Q2)

**v3.1 wording problem:** "Judge gate MUST use a non-fleet cloud
model" was imprecise. The threat is shared blast radius (poisoned
model image), not cloud-vs-local locality.

**Redesign principle: independence, not cloud-vs-local.** The judge
gate must satisfy:

1. **Image-independence** from the work it's judging — different
   model family, different distribution channel, different supply
   chain.
2. **Provenance-verified** — vendor-attested cloud endpoint, signed
   Ollama manifest, locally-built-from-source weights, or
   hardware-rooted attestation.
3. **Quorum on governance-critical operations** — `CONSULTANT_CLOSEOUT`
   validation and rule-coverage Stage-2 gating require **≥2
   independent judges** to agree.

**Three orthogonal axes** (separated; v3.1 conflated them):

- Cost: free / paid.
- Locality: local-tailnet / remote-cloud.
- Provenance: unverified / vendor-attested / source-built /
  hardware-rooted.

A judge gate cares only about the provenance axis. The fleet —
including Tailscale Ollama hosts — can produce verified-provenance
judges if the operator builds from attested-source weights and the
Ollama manifest is signed.

**Gate-type map:**

| Gate type | Acceptable judges |
|---|---|
| Routine routing (no governance impact) | Any 1 fleet model |
| Bundle integrity verification | SLSA verifier + Cosign (deterministic, no LLM) |
| Rule-coverage Stage-1 (every build) | Deterministic keyword check, no LLM |
| Rule-coverage Stage-2 (weekly) | Quorum of 2, different families |
| `CONSULTANT_CLOSEOUT` validation | Quorum of 2, different families, ≥1 verified-provenance |

A fully-fleet quorum (qwen-3-235b on Cerebras + llama-3.3-70b on
Groq, both vendor-attested + different families) satisfies the gate
at zero token cost. Paid Sonnet is the easiest verified-provenance
judge but not the only path.

**New module:** `scripts/global/judge-quorum.js` (#895) handles
2-judge agreement with disagreement-escalation.

## 4. Failover & redundancy map (Q3)

| Layer | Failure mode | Failover / redundancy |
|---|---|---|
| Bundle distribution | CF Worker / R2 down | 3-tier: R2 primary → GitHub release asset CDN → npm-bundled snapshot baked into install. All three serve the same content-addressed bundle hash. |
| Routing layer | Cloud provider rate-limited | Header-driven spillover (HAMR child 9). Anthropic Batch API used for non-time-critical work. |
| Fleet routing | One Ollama host down | `cascade-dispatch.js` already routes around failed hosts; substrate-health probe (HAMR child 2) marks degraded hosts and the router reads it. |
| Identity / signing | Hardware enclave unavailable | T1 → T2 → T3 → T4 ladder (R1). |
| Judge gate | One judge family unavailable | Quorum is "2 of N" across 5+ families. Loss of any single provider doesn't break the gate. |
| Wiki retrieval | MCP server down | `rag-search.js` already has ripgrep fallback to local wiki. |
| Mailbox (R2) | R2 unavailable | Local SQLite WAL queue at `~/.megingjord/mailbox-outbox.db`; flushes when R2 returns. Receivers poll `gh` API as side-channel during outage. |
| Cache (Anthropic) | Cache miss / 5-min eviction | Bundle is content-addressed by SHA-256; rebuild is deterministic and idempotent. Cache miss costs +25% on the affected call. |
| Capability probes | Probe times out | All 6 HAMR probes are time-bounded (≤5 s) and fail-soft. Probe failure never blocks the harness. |
| CF account compromised | Whole substrate hostile | DC-2 (SLSA verifier) + R3 (signed bundle) + R2 (signed envelopes) + R1 (signed handoffs) form 4-layer defense in depth. A compromised CF account cannot forge a valid SLSA attestation, signed envelope, or signed handoff without operator keys. |

**Critical guarantee:** the existing harness (cascade-dispatch +
free-router + capability-probe + wiki tooling — all REUSE/REFACTOR
per S1 #876) **never gets worse** when HAMR is added. Tier 3 means
HAMR is invisible; the operator continues working with v3.0-
equivalent capability.

**Remaining genuine fragility points** (bounded, not single-point
failures):

- PKCE in `workers-oauth-provider` beta (S6/A5-S MEDIUM): pre-1.0;
  HAMR's own integration tests + version pin + Dependabot alerts.
- Transitive npm dependencies (S6/A4-T MEDIUM): SLSA covers the
  harness package, not its transitive `npm install` graph. `npm
  audit signatures` + Dependabot + minimal-dependency policy.

## 7. Residual risks after v3.2

3 of 9 STRIDE MEDIUM/HIGH cells remain (down from 9 in S6 #881):

- **A2-S MEDIUM (DPoP key, T4 only):** ephemeral CI runners cannot
  use hardware enclave. Bounded by 4-h JWT TTL on T4.
- **A4-T MEDIUM (transitive npm deps):** SLSA covers harness; not
  transitive `npm install` graph. Bounded by Dependabot + `npm
  audit signatures`.
- **A5-S MEDIUM (PKCE in workers-oauth-provider beta):** library
  pre-1.0. Bounded by HAMR integration tests + version pin.

A3-E HIGH (poisoned fleet model fabricates closeout) is **eliminated**
by R1 (signed artifacts) + R2 (signed envelopes) + R3 (signed
bundle) + judge-quorum independence requirement. The other 6
MEDIUM/HIGH cells are also eliminated by R1–R3.
